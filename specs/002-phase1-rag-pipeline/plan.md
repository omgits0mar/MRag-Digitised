# Implementation Plan: Phase 1 — RAG Foundation Pipeline

**Branch**: `002-phase1-rag-pipeline` | **Date**: 2026-04-07 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/002-phase1-rag-pipeline/spec.md`
**Constitution Version**: 1.0.0

## Summary

Build the complete Phase 1 foundation for the MRAG platform: a data processing pipeline that ingests, chunks, enriches, and exports the Natural Questions dataset; a multilingual embedding system with FAISS vector indexing; and a basic retrieval pipeline that accepts natural language queries and returns ranked, metadata-enriched results. This constitutes the end-to-end data backbone from raw dataset to searchable knowledge base.

## Technical Context

**Language/Version**: Python 3.10+ (established in Feature 000)
**Primary Dependencies**: Pandas 2.2.3, NumPy 2.2.6, sentence-transformers 4.1.0, faiss-cpu 1.11.0, Pydantic 2.11.3, structlog 25.2.0
**Storage**: FAISS index files (`.faiss`) + JSON metadata files (no database in Phase 1)
**Testing**: pytest 8.3.5, pytest-asyncio 0.26.0, pytest-cov 6.1.1, scikit-learn 1.6.1 (for evaluation metrics)
**Target Platform**: Local development machine (Linux/macOS), CPU-only (GPU optional)
**Project Type**: Library (Python package) — CLI pipeline runners with programmatic API
**Performance Goals**: <200ms retrieval latency for K=10, <30 min full dataset processing, <5s index load
**Constraints**: Multilingual (50+ languages), offline-capable after dataset download, ~86K records
**Scale/Scope**: ~86K question-answer pairs (Natural-Questions-Filtered.csv), ~86K-200K chunks after splitting, single-node FAISS
**Dataset Schema**: 3 columns — `question` (str), `short_answers` (str), `long_answers` (str)

## Technical Approach

### Module 1: Data Processing (`src/mrag/data/`)

**Ingestion** (`ingestion.py`):
- Load CSV/JSON using `pandas.read_csv()` / `pandas.read_json()`
- Dataset: `Natural-Questions-Filtered.csv` with columns `question`, `short_answers`, `long_answers` (~86K rows)
- Validate each row against `RawRecord` Pydantic model (fields: `question`, `short_answers`, `long_answers`)
- Returns `tuple[list[RawRecord], int]` — validated records and skipped count
- Skip and log invalid records; raise `DataProcessingError` only if 100% of records fail
- Normalize text: strip whitespace, normalize Unicode (NFC), handle encoding

**Chunking** (`chunking.py`):
- `TextChunker` class with configurable `chunk_size` (tokens) and `chunk_overlap` (tokens)
- Tokenization: split on whitespace for token counting (language-agnostic)
- Sentence splitting: regex-based on `.!?` + Unicode sentence terminators, with lookahead to handle abbreviations
- Sliding window: advance by `chunk_size - chunk_overlap` tokens per step, snap to nearest sentence boundary
- Edge cases: text shorter than chunk_size → single chunk; empty text → raise error

**Enrichment** (`enrichment.py`):
- Question type classifier: regex patterns on question prefix
  - `Who/What/When/Where/Which + ...?` → factoid
  - `How many/How much` → factoid
  - `List/Name (all|some|the)` → list
  - `Is/Are/Was/Were/Do/Does/Did/Can/Could + ...?` → yes_no
  - `Explain/Describe/Why/How (does|do|is|are)` → descriptive
  - Default → unknown
- Domain classifier: keyword-count matching against predefined domain keyword lists (science, history, geography, health, technology, mathematics), classify by highest hit count; defaults to "general"
- Difficulty: `easy` if short answer present, `medium` if long-only, `hard` if descriptive question with long answer >100 tokens
- `source_id`: deterministic MD5 hash of `question:short_answers:long_answers` (16 hex chars) — satisfies FR-005

**Export** (`export.py`):
- Serialize `ProcessedDocument` to JSON, write one per line to `.jsonl`
- Deterministic split: sort by `doc_id`, use MD5 hash of `doc_id:seed` — `hash % 100 < split_ratio * 100` assigns to train (streaming-compatible, no dependency on record count)
- Log statistics: total records, train count, eval count, total chunks

### Module 2: Embeddings & Indexing (`src/mrag/embeddings/`)

**Encoder** (`encoder.py`):
- Wrap `sentence_transformers.SentenceTransformer` with configured model name
- `encode()`: batch-process texts, L2-normalize output via `sklearn.preprocessing.normalize`
- `encode_single()`: convenience for single query embedding
- Lazy model loading: initialize model on first `encode()` call
- Log: model loaded, batch progress, total encoding time

**Indexer** (`indexer.py`):
- `FAISSIndexer` wraps `faiss.IndexFlatIP` (inner product = cosine on L2-normed vectors)
- `build_index()`: create fresh index, add all vectors
- `search()`: query with `index.search()`, return scores and integer IDs
- `save()`/`load()`: use `faiss.write_index()` / `faiss.read_index()`
- Validate dimension on build and search

**Metadata Store** (`metadata_store.py`):
- Python `dict[int, MetadataEntry]` in memory
- `add()`: insert by FAISS integer ID
- `get()`: O(1) lookup by ID
- `filter()`: linear scan with field-value match (acceptable for <1M entries)
- `save()`/`load()`: `json.dump()` / `json.load()` with custom encoder for Pydantic models

**Pipeline** (`pipeline.py`):
- Load train JSONL from data pipeline
- Extract all chunk texts + build metadata entries
- Batch-encode via `EmbeddingEncoder`
- Build FAISS index via `FAISSIndexer`
- Populate `MetadataStore`
- Save all artifacts to disk
- Log: chunks processed, index size, encoding time, total time

### Module 3: Retrieval (`src/mrag/retrieval/`)

**Models** (`models.py`):
- `RetrievalRequest`: Pydantic model with `query`, `top_k`, `score_threshold`, `metadata_filters`
- `RetrievalResult`: Pydantic model with all output fields (chunk_text, scores, metadata)

**Ranking** (`ranking.py`):
- `rerank()` function with configurable `alpha` (default 0.7)
- `relevance_score = alpha * cosine_similarity + (1 - alpha) * metadata_boost`
- Metadata boost computation:
  - +0.1 if `has_short_answer` (answer availability signal)
  - +0.1 if question_type matches query pattern (e.g., factoid query → factoid result)
  - Boost capped at 1.0
- Sort by relevance_score descending

**Retriever** (`retriever.py`):
- `RetrieverService` takes encoder, indexer, metadata_store as constructor dependencies
- `retrieve()` method orchestrates the full flow
- Over-fetch: if `score_threshold` is set, request `top_k * 2` from FAISS to account for post-filtering
- Metadata filter: post-filter results before re-ranking
- Return sorted, filtered results capped at `top_k`

## Key Technical Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Embedding model | `paraphrase-multilingual-MiniLM-L12-v2` (384 dims) | 50+ languages, CPU-feasible, per Article III |
| FAISS index | `IndexFlatIP` (exact search) | ~86K vectors, accuracy > speed per Article IV |
| Vector normalization | L2-normalize before indexing | Inner product on unit vectors = cosine similarity |
| Chunking | Sliding window, sentence-boundary aligned | Semantic coherence per Article II |
| Dataset | `Natural-Questions-Filtered.csv` | 86K rows, 3 columns: question, short_answers, long_answers |
| Metadata storage | JSON dict keyed by int | O(1) lookup, no DB dependency, human-readable |
| Re-ranking | Weighted linear combination (alpha=0.7) | Beyond raw cosine per Article IV, interpretable |
| Export format | JSONL with MD5-hash 90/10 split | Streaming-compatible, deterministic per Article II |
| Data models | Pydantic v2 models for validation | Type safety at boundaries per Article IX |
| Question classification | Regex-based heuristics | Deterministic, no ML dependency, offline-capable |
| Domain classification | Keyword-count matching | Simpler than TF-IDF, deterministic, sufficient for 6 domains |
| Batch size | 64 for embeddings | Memory/throughput balance on 16GB RAM |
